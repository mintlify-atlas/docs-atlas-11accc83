The `tonic::server` module contains the traits and types that service implementations depend on. Code generated by `tonic-build` produces a trait with the RPC method signatures; you implement that trait, and tonic handles the HTTP/2 framing, codec, and routing. This page covers the most important server-side primitives: `NamedService`, the service traits, request introspection, and streaming responses.

## `NamedService` trait

Every gRPC service must have a statically known name used for routing incoming requests on the server's path (`/ServiceName/MethodName`).

```rust
pub trait NamedService {
    /// The `Service-Name` as described in the gRPC HTTP/2 protocol.
    const NAME: &'static str;
}
```

Generated server stubs implement this automatically, using the fully qualified protobuf service name (e.g., `helloworld.Greeter`). If you are building a service by hand you must implement it yourself:

```rust
impl tonic::server::NamedService for MySvc {
    const NAME: &'static str = "mypackage.MyService";
}
```

<Note>
`Server::add_service` requires both `NamedService` and `Clone + Send + Sync + 'static`. The name is used to match the `:path` pseudo-header on incoming HTTP/2 requests.
</Note>

## Service traits

The `tonic::server` module re-exports four service traits that generated code relies on. You generally implement the generated trait rather than these directly.

```rust
pub use self::service::{
    UnaryService,
    ClientStreamingService,
    ServerStreamingService,
    StreamingService,
};
```

| Trait | Request | Response |
|---|---|---|
| `UnaryService` | `Request<T>` | `Response<U>` |
| `ClientStreamingService` | `Request<Streaming<T>>` | `Response<U>` |
| `ServerStreamingService` | `Request<T>` | `Response<Pin<Box<dyn Stream<Item=Result<U,Status>>>>>` |
| `StreamingService` | `Request<Streaming<T>>` | `Response<Pin<Box<dyn Stream<Item=Result<U,Status>>>>>` |

All four are `async_trait` compatible and return `Result<Response<…>, Status>`.

## Accessing request metadata and remote address

Inside a service handler the `Request<T>` parameter carries metadata, connection information, and HTTP extensions.

### Metadata

```rust
async fn say_hello(
    &self,
    request: Request<HelloRequest>,
) -> Result<Response<HelloReply>, Status> {
    // Read a custom header attached by the client.
    let token = request
        .metadata()
        .get("authorization")
        .and_then(|v| v.to_str().ok());

    // ...
}
```

<ParamField path="metadata" type="fn(&self) -> &MetadataMap">
  Returns the gRPC metadata (HTTP/2 headers) sent by the client. Header names follow the gRPC naming convention (lowercase, no leading colon).
</ParamField>

<ParamField path="metadata_mut" type="fn(&mut self) -> &mut MetadataMap">
  Mutable access to request metadata. Typically used in interceptors, not service handlers.
</ParamField>

### Remote address

```rust
// Requires the "server" feature (enabled by default).
let addr: Option<std::net::SocketAddr> = request.remote_addr();
```

<ParamField path="remote_addr" type="fn(&self) -> Option<SocketAddr>">
  Returns the peer address of the client connection. Returns `None` when the underlying IO type does not implement `Connected`, for example on Unix domain socket connections.
</ParamField>

<ParamField path="local_addr" type="fn(&self) -> Option<SocketAddr>">
  Returns the local address the server accepted the connection on. Also returns `None` for non-TCP transports.
</ParamField>

<ParamField path="peer_certs" type="fn(&self) -> Option<Arc<Vec<CertificateDer<'static>>>>">
  Returns the peer's TLS certificates for mutual TLS setups. Only available when the `_tls-any` feature is enabled and the connection was established with TLS.
</ParamField>

## Returning `Response<T>`

Service methods return `Result<Response<T>, Status>`. Construct a `Response` with `Response::new`:

```rust
Ok(Response::new(HelloReply {
    message: format!("Hello, {}!", request.into_inner().name),
}))
```

You can also attach trailing metadata to a response before returning it:

```rust
let mut response = Response::new(HelloReply {
    message: "Hello!".into(),
});
response
    .metadata_mut()
    .insert("x-request-id", "abc123".parse().unwrap());
Ok(response)
```

## `tonic::Status` as the error type

Service methods return `Err(Status)` to signal a gRPC error to the client. See the [Status API reference](/api/status) for all constructors and code values. Common patterns:

```rust
// Not found
return Err(Status::not_found("user not found"));

// Invalid input
return Err(Status::invalid_argument("name must not be empty"));

// Unimplemented method
return Err(Status::unimplemented("not yet available"));

// Propagate an internal error
return Err(Status::internal(format!("database error: {}", e)));
```

<Warning>
Never return sensitive information in the `Status` message — it is transmitted to the client as plain text in the `grpc-message` trailer header.
</Warning>

## Streaming responses

Server-streaming and bidirectional-streaming RPCs return a `Response` whose inner type is a pinned `Stream`. The idiomatic way is to use `tokio_stream` or `async-stream`.

### Server streaming

```rust
use tokio_stream::wrappers::ReceiverStream;
use tokio::sync::mpsc;

type ResponseStream = Pin<Box<dyn Stream<Item = Result<Feature, Status>> + Send>>;

async fn list_features(
    &self,
    request: Request<Rectangle>,
) -> Result<Response<ResponseStream>, Status> {
    let (tx, rx) = mpsc::channel(128);

    tokio::spawn(async move {
        for feature in features_in_rect(request.into_inner()) {
            tx.send(Ok(feature)).await.unwrap();
        }
    });

    Ok(Response::new(Box::pin(ReceiverStream::new(rx))))
}
```

### Bidirectional streaming

```rust
async fn route_chat(
    &self,
    request: Request<Streaming<RouteNote>>,
) -> Result<Response<ResponseStream>, Status> {
    let mut inbound = request.into_inner();

    let output = async_stream::try_stream! {
        while let Some(note) = inbound.message().await? {
            yield RouteNote {
                location: note.location,
                message: format!("echo: {}", note.message),
            };
        }
    };

    Ok(Response::new(Box::pin(output)))
}
```

The return type for streaming responses is always:

```rust
Pin<Box<dyn Stream<Item = Result<ResponseType, Status>> + Send + 'static>>
```

<Tip>
The `async-stream` crate's `try_stream!` macro is the most ergonomic way to produce a `Stream` from async code. Add `async-stream = "0.3"` to your `Cargo.toml`.
</Tip>
